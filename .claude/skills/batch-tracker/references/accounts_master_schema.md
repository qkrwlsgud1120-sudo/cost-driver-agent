# accounts_master.json 스키마

Phase 0에서 생성되고, 이후 모든 배치 처리에서 참조·갱신되는 중앙 상태 파일이다.

**대분류 도입(§0) 이후 구조**: 공통/특정 판단, 4-type 분류, 원가동인 추천, 회계사확정은 이제 **계정코드가 아니라 대분류 단위**로 이루어진다. `categories` 섹션이 대분류별 "살아있는 상태"(분류·추천·확정)를 담고, `accounts`의 각 계정코드는 `대분류` 필드로 그 상태를 참조한다. 세부계정이 대분류 확정값을 그대로 따르면 "원칙준용", 개별적으로 다른 원가동인을 쓰면 "예외지정"이며 이때만 `예외확정`에 독립된 상태가 저장된다.

**세부계정 중복 통합**: 이 데이터셋의 계정코드는 전사 공통 코드 체계라서 계정코드↔계정명이 1:1이고, 하나의 계정코드가 여러 부서에 걸쳐 동일하게 등장한다. 따라서 `accounts`의 각 계정코드는 **부서 무관 단일 레코드**다 — `적용방식`/`예외확정`은 부서별 dict가 아니라 계정코드당 하나의 값이며, 세부계정 하나를 예외 지정하면 그 계정이 등장하는 모든 부서에 동일하게 적용된다.

## `categories[대분류]` 필드 설명
| 필드 | 설명 |
|---|---|
| `구분` | `공통`(2개 이상 부서에 이 대분류의 세부계정이 등장) 또는 `특정`(1개 부서만). `account_segmentation.json`의 배정과 항상 일치해야 한다 |
| `카테고리분류상태` | `미착수` \| `분류중` \| `추가판단필요(검토대기)` \| `분류완료` |
| `추가판단필요여부` | true면 이 대분류의 four_type이 아직 확정되지 않았다는 뜻. "세부계정 성격이 이질적이라 하나의 원가동인으로 묶기 부적절한 경우"도 여기 포함된다(account-classifier.md 참조). Path 1/Path 2로 해소되면 false로 바뀐다 |
| `추천상태` | `미착수` \| `추천중` \| `검증중` \| `완료` |
| `원가동인.recommended_drivers` | 대분류 전용 driver-recommender 호출로 생성된 후보 배열(세부계정 추천 재활용 아님). 1~3개, `rank` 오름차순, 각 원소는 `rank`/`driver`/`reason`/`근거출처` |
| `회계사확정` | **부서 무관 단일 객체**(계정코드 체계와 달리 부서별 dict가 아님 — 대분류 확정은 원칙적으로 전 부서 공통 적용이며, 부서마다 다르게 하고 싶으면 그 세부계정을 예외지정한다). `확정여부`(`미확정`\|`승인`\|`수정`, 추가판단 필요 대분류는 `미확정`\|`확정`), `확정원가동인`, `확정순위`, `확정일시`로 구성 |
| `처리경로` | `AI추천-확정` \| `AI추천-확정(대기)`(Path 1, 4-type만 확정돼 원가동인 재추천 대기) \| `사람직접확정`(Path 2) |
| `판단경로` | 이 대분류의 four_type/원가동인이 **누구/무엇의 판단으로 채워졌는지**. `AI 판단`(`track_batch.py complete()` — 실제 account-classifier/driver-recommender 서브에이전트 배치 결과) \| `규칙 기반 폴백`(`streamlit_app/phase1_apply.py` — 서브에이전트 없이 로컬 정규식 규칙으로 채운 PoC 대체 경로. 이미 `판단경로: "AI 판단"`인 대분류는 폴백이 덮어쓰지 않는다) \| `사람 직접 확정`(추가판단 필요 대분류를 회계사가 Path 2로 직접 확정한 경우). 최종 엑셀(`write_results.py`)의 "판단 경로" 컬럼과 데모 요약 리포트의 "AI 판단 vs 규칙 기반 폴백" 통계가 이 필드를 그대로 쓴다 |
| `처리완료` | 재개(resume)의 1차 기준. Phase 1(대분류 단위 account-classifier→driver-recommender→result-validator) 완료 시 true |
| `인용출처_경고` | **선택 필드**(정상이면 존재하지 않음). `track_batch.py complete()`가 `recommended_drivers`/`세부계정_설명` 중 `근거출처: "문서인용"`인 항목의 인용문에서 `cost-driver-framework` 참조 문서 화이트리스트(`category_utils.REFERENCE_DOCS`) 밖의 `.md`/`.json` 파일명이 언급된 걸 발견하면 그 목록을 담는다. result-validator를 이미 통과한 값이라 병합을 막지는 않고 경고만 남긴다 — 회계사 확정 전에 검토를 권장한다 |

## `accounts[계정코드]` 필드 설명
| 필드 | 설명 |
|---|---|
| `대분류` | 이 계정이 속한 대분류명. `category_utils.derive_category()`로 계정명에서 도출(하이픈 앞부분, 없으면 전체) |
| `등장부서` | 이 계정코드가 실제 등장한 부서 목록(전사 공통 코드 체계라 여러 부서에 걸쳐 동일 코드가 등장할 수 있음) |
| `구분` | 이 세부계정 자신의 등장부서 수 기준(`min_depts` 이상이면 공통, 미만이면 특정). **소속 대분류의 `구분`과 다를 수 있다** — 대분류가 공통이어도 그 아래 특정 세부계정 하나는 1개 부서에만 등장할 수 있다(예: 대분류 "전산비"는 공통이지만 그 아래 "전산비-네트웍운영용역"은 IT운영팀에만 존재해 구분="특정") |
| `적용방식` | **부서 무관 단일 값**. `"원칙준용"`(기본값 — 대분류 확정값을 그대로 상속) 또는 `"예외지정"`(이 계정만 개별 확정값 사용). 예외지정하면 `등장부서`에 열거된 모든 부서에 동일하게 적용된다 |
| `예외확정` | `적용방식`이 `"예외지정"`일 때만 값이 채워지는 **단일 객체**(부서별 dict 아님). 형태는 `categories[대분류]`와 동일(four_type/분류근거/추가판단필요여부/원가동인/회계사확정/처리경로/**판단경로**/처리완료) — 예외지정은 항상 회계사가 직접 채우는 값이라 `판단경로`는 `"사람 직접 확정"`으로 고정된다 |
| `설명` | 이 세부계정이 통상 어떤 성격의 비용인지 설명하는 객체 `{"내용": str, "근거출처": "문서인용"\|"일반적인 회계 관행 기반 추정"}`. account-classifier가 대분류 분류와 함께 생성한다(§ 세부계정 설명 참조). 아직 생성 전이면 `null` |

**이 계정코드의 최종 유효 상태는 항상 `category_utils.resolve_account_effective(master, code)`로 계산한다** — 원칙준용이면 `categories[대분류]`, 예외지정이면 `accounts[code].예외확정`을 반환한다(부서 무관 — 계정코드 하나가 등장하는 모든 부서에 동일하게 적용). track_batch.py, app.py, write_results.py 등 모든 소비자가 이 함수 하나로 통일해서 쓴다(직접 필드를 읽지 않는다).

> **레거시 필드 안내**: Phase 0(`scan_accounts.py`)가 생성하는 계정 스켈레톤에는 `분류상태`/`four_type`/`분류근거`/`추가판단필요여부`/`추천상태`/`원가동인`/`부서별_비고`/`회계사확정`/`처리경로`/`처리완료` 필드가 계정코드 레벨에도 기본값으로 남아있을 수 있다. 대분류 도입 이후 이 필드들은 **사용되지 않는다** — 실제 분류·추천·확정 상태는 반드시 위 `resolve_account_effective()` 경로로만 조회한다.

## 추가판단 필요 해소 경로 (Path 1 vs Path 2) — 대분류 단위
- **Path 1**: 회계사가 대분류의 4-type만 확정 → `categories[대분류].four_type`/`추가판단필요여부`만 갱신, `처리경로`="AI추천-확정(대기)", `원가동인`은 아직 null → 대분류 단위로 driver-recommender 재호출 필요.
- **Path 2**: 회계사가 대분류의 4-type과 원가동인을 함께 확정 → `categories[대분류]`를 한 번에 갱신, `처리경로`="사람직접확정" → result-validator만 재실행.
- 세부계정을 예외지정하는 경우도 동일한 Path 1/Path 2 구분이 `accounts[code].예외확정` 안에서 그대로 적용된다.

## 재개(resume) 규칙
처리가 중단되면 `categories[대분류].처리완료: false`인 대분류만 다시 처리한다. 세부계정의 실질 처리 여부는 `resolve_account_effective()`가 반환하는 `처리완료`로 판단한다(원칙준용이면 대분류를, 예외지정이면 그 계정 자체의 값을 본다).
