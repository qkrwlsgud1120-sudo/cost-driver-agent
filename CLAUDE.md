# 원가동인(Cost Driver) 추천 에이전트 — 오케스트레이터 지침

> 이 문서는 메인 Claude 세션의 행동 규칙이다. 상세 설계 배경은 `/원가동인_에이전트_설계서 (1).md` 참조.

## 0-1. 대분류(카테고리) 도입 — 판단 단위 변경

**공통/특정 판단, 4-type 분류, 원가동인 추천, 회계사확정의 단위는 계정코드가 아니라 대분류다.** 대분류는 계정명에서 도출한다(하이픈 앞부분, 없으면 계정명 전체 — `category_utils.derive_category()`). 동일 대분류가 2개 이상 부서에 등장하면 공통 대분류, 1개 부서만이면 특정 대분류다.

- 대분류를 확정하면 그 아래 세부계정(계정코드) 전체가 자동으로 "원칙 준용" 상태가 되어 대분류 확정값을 상속한다.
- 특정 세부계정만 다르게 하고 싶으면 그 계정을 "예외 지정"해 개별 확정한다(대분류 확정을 기다릴 필요 없이 먼저 지정 가능).
- **세부계정(계정코드)은 전사 공통 코드 체계라 여러 부서에 걸쳐 동일하게 등장한다** — 부서 조합이 아니라 계정코드 단위로 중복 없이 다루며, 예외 지정하면 그 계정이 등장하는 모든 부서(`accounts[code].등장부서`)에 동일하게 적용된다. 확정 UI·최종 엑셀 모두 이 계정코드 하나당 한 번만 표시한다(등장부서는 "N개 부서 공통" 요약 또는 목록으로 표시).
- account-classifier는 대분류 4-type 분류와 함께, 그 아래 세부계정마다 어떤 성격의 비용인지 설명(`세부계정_설명`)도 함께 생성해 `accounts[code].설명`에 저장한다. 확정 UI와 최종 엑셀("계정 설명" 컬럼)에 노출한다.
- account-classifier·driver-recommender 둘 다 이제 **대분류 단위**로 호출한다(세부계정 단위 아님). 자세한 스키마는 `.claude/skills/batch-tracker/references/accounts_master_schema.md`, 판단 절차는 `.claude/agents/account-classifier.md`·`driver-recommender.md` 참조.
- 원칙준용/예외지정 판정 로직은 `category_utils.resolve_account_effective()` 한 곳에만 있다 — track_batch.py, app.py, write_results.py 모두 이 함수를 통해서만 최종 확정값을 읽는다.

아래 §2~§6의 "계정" 관련 서술은 이 원칙에 따라 "대분류(공통/특정 판단·4-type·원가동인 추천 단위)"와 "세부계정(계정코드, 부서 무관 단일 레코드 — 원칙준용/예외지정 단위)"으로 나눠 읽는다.

## 0. 역할 정의

메인 세션은 **오케스트레이터**다. 계정 분류·원가동인 추천·결과 검증은 절대 메인 세션이 직접 판단하지 않는다 — 반드시 아래 서브에이전트에게 Task로 위임한다.

메인 세션이 직접 수행하는 것:
- 코드(규칙 기반) 단계: 계정 인벤토리 스캔 실행 지시, 공통/특정계정 분리(Phase 0.5) 1차 판별 및 LLM 재확인 결과 반영, 입력 검증, 배치 식별, 병합/출력, 배치 로그 갱신, 회계사 확정 결과 반영 (모두 스킬 스크립트 실행)
- 서브에이전트 호출 순서 관리 및 산출물 검증
- 에스컬레이션 판단 및 사용자 확인
- ⑧ 데모 요약 리포트 작성 (서브에이전트 아님, 메인 세션이 직접 작성)

## 1. 실행 트리거

- 슬래시 커맨드 없음. 사용자가 자연어로 요청("이 파일 분류해줘", "OO부서 파일 추가해줘", "검토 완료한 파일 재업로드했어")하면 워크플로우 시작으로 해석한다.
- `/output/accounts_master.json` 존재 여부로 Phase 0 완료 여부를 판단한다.
  - 없음 → Phase 0부터 시작. 단, `/input/departments/`에 **전체 부서 파일**이 모두 준비되었는지 먼저 사용자에게 확인한다. 일부만 있으면 진행하지 않고 누락 부서를 알려준다. (Phase 0 + Phase 0.5 규칙기반 분리까지는 `/streamlit_app/app.py`의 사이드바 업로드 UI에서 직접 실행할 수도 있다 — 부서 개수/이름은 고정값이 아니라 업로드된 파일에서 그때그때 동적으로 인식된다. 단 그 뒤 Phase 1(4-type 분류·원가동인 추천)은 서브에이전트 호출이 필요해 Streamlit 혼자서는 실행할 수 없고, 사용자가 다시 이 세션으로 돌아와 요청해야 한다.)
  - 있음 → `/output/account_segmentation.json` 존재 여부 + `needs_llm_recheck` 배열이 비어있는지로 Phase 0.5 완료 여부를 판단한다.
    - 미완료(파일 없음 또는 `needs_llm_recheck`에 항목 존재) → Phase 1을 시작하지 않고 Phase 0.5부터 진행/재개한다.
    - 완료 → 신규 배치 추가인지, 기존 배치 갱신인지, 피드백 재반영(추가판단 필요 계정 4-type 직접 기입 재업로드, 또는 확정 컬럼을 채운 최종 엑셀 재업로드)인지 사용자에게 확인 후 Phase 1로.

## 2. 워크플로우 (Phase 0 → Phase 0.5 → 배치별 Phase 1 → 회계사 확정 피드백)

**Phase 0 (프로젝트당 1회)**
- ⓪ 계정 인벤토리 스캔 — `batch-tracker` 스킬의 `scan_accounts.py` 실행 (코드, LLM 미사용) → `/output/accounts_master.json` 생성. 이 스크립트는 `/input/departments/` 안의 모든 파일을 스캔해 등장하는 부서명을 그대로 인식하므로 부서 개수·이름을 코드 어디에도 하드코딩하지 않는다. Streamlit 앱(`streamlit_app/app.py`) 사이드바에서 부서 파일을 업로드하고 "분석 시작"을 눌러도 동일한 `scan()` 함수가 그대로 실행된다 — 이 경우 기존 `/output/`과 `/input/departments/`는 자동으로 `output_backup_{timestamp}/`에 백업된 뒤 새로 시작한다(기존 분류·추천·확정 데이터 유실 방지).

**Phase 0.5 (프로젝트당 1회, Phase 0 직후 · Phase 1 이전 — 완료 전까지 Phase 1 시작 금지)**

전체 부서 계정을 부서 배치 단위가 아니라 **통합**으로 먼저 살펴, 어떤 계정이 여러 부서에 걸친 "공통계정"이고 어떤 계정이 "특정계정"인지 확정한다. 이 확정이 끝나야 Phase 1에서 공통 계정에 통일 원가동인을 적용할 수 있다.

1. ⓪.5-1 규칙 기반 1차 분리 (코드) — `batch-tracker` 스킬의 `segment_accounts.py analyze` 실행. 대분류(계정명에서 도출)가 2개 이상 부서에 등장하면 공통 후보, 1개면 특정으로 분리한다. 이 스크립트가 동시에 `accounts_master.json`에 `대분류`/`구분` 필드와 `categories` 스켈레톤을 병합 반영한다(재실행해도 기존 대분류의 분류·추천·확정 상태는 덮어쓰지 않음).
2. ⓪.5-2 추가판단 필요 사례 LLM 재확인 — `needs_llm_recheck`가 비어있지 않으면, 해당 대분류들을 Task로 **account-classifier**에게 "Phase 0.5 재확인 모드"(`account-classifier.md` 참조)로 위임한다. "공통유지" 또는 "특정전환" 판정과 사유를 받는다.
3. ⓪.5-3 판정 반영 (코드) — `segment_accounts.py apply-llm`으로 LLM 판정을 `account_segmentation.json`에 병합하고, `특정전환` 판정을 받은 대분류는 `accounts_master.json`의 `categories[대분류].구분`도 `특정`으로 동기화한다.
4. ⓪.5-4 시각화 산출 (코드) — `excel-io` 스킬의 `write_segmentation_excel.py`(입력: `account_segmentation.json` + `accounts_master.json`)로 `/output/account_segmentation.xlsx`(시트1 공통대분류, 시트2 특정대분류 부서별 그룹, 시트3 세부계정 전체 목록 — 읽기 전용 참고본)를 생성한다. 실제 확정 UI는 `/streamlit_app/app.py`(Streamlit 앱)이며, 이 파일의 `build_confirm_data()`를 그대로 import해서 재사용한다. `/docs/demo_summary_{batch_id}.md`에도 이 분리 결과 요약을 포함한다(⑧ 참조).
5. `needs_llm_recheck`가 완전히 비워질 때까지 2~3을 반복한다. 비워지면 Phase 0.5 완료 — Phase 1 진행 가능.

**Streamlit 확정 앱(`streamlit run streamlit_app/app.py`)**: "전체" 탭에서 공통계정·특정계정(추가판단 필요 계정 포함)을 확정하면, 부서별 탭에 자동으로 반영된다. 부서 탭은 `account_segmentation.json`에서 실제로 인식된 부서 목록으로 그때그때 동적 생성되며(개수·이름 고정 아님), 공통계정은 한 번만 확정하면 `등장부서`에 포함된 모든 부서 탭에 동일하게 나타나므로 부서마다 같은 계정을 중복 판단하는 문제가 애초에 생기지 않는다. 부서 탭은 확인 전용이며 재입력을 받지 않는다. 진행률 표시(`st.progress`)는 AI 추천 존재 여부가 아니라 실제 사람의 승인/직접입력만 카운트한다. 앱 내에서 "전체 저장"을 눌러야 화면상의 확정 초안이 `accounts_master.json`과 부서별 최종 엑셀에 실제로 반영된다 — 그 전까지는 세션 상태에만 있는 초안이다. 반영·엑셀 동기화 세부 절차는 아래 "회계사 확정 경로 B"를 따른다. 이 앱은 사이드바에서 새 부서 파일 업로드 → Phase 0(스캔) + Phase 0.5(규칙기반 공통/특정 분리)까지도 직접 실행할 수 있다(§1 참조) — 다만 Phase 0.5의 LLM 재확인과 Phase 1(4-type 분류·원가동인 추천)은 서브에이전트 호출이 필요해 이 화면 혼자서는 끝낼 수 없다.

**Phase 1 (대분류 배치마다 반복 — 배치는 부서 묶음 단위로 나눠도 되고, 대분류 단위 처리 자체는 부서 배치와 독립적이다)**
1. ① 입력 검증 (코드) — 실패 시 자동 재시도(형식 안내), 반복 실패 시 에스컬레이션
2. ② 배치 식별 (코드) — `accounts_master.json` 기준 신규/갱신 판별
3. ③ 대분류 특성 분석 — Task로 **account-classifier** 호출 (대분류 단위, 세부계정 목록 전체를 맥락으로 참고)
4. ④ 분류 결과 검증 및 분기 (규칙 + LLM 자기검증) — 추가판단 필요 대분류는 회계사 판정 대기로 분리, 나머지는 ⑤로 진행
5. ⑤ 원가동인 추천 — Task로 **driver-recommender** 호출 (대분류 단위, 세부계정 개별 추천 재활용 안 함). 원가동인 후보는 **최대 3순위까지**, 순위마다 개별 근거를 붙여 제시한다(억지로 3개를 채우지 않음 — `recommended_drivers` 배열, `account-classifier.md`가 아니라 `driver-recommender.md` 참조). **공통 대분류**(Phase 0.5에서 확정)는 전 부서 통일 원가동인 원칙에 따라 `accounts_master.json`에 이미 확정된 추천(`categories[대분류].원가동인.recommended_drivers`)이 있으면 그것을 우선 채택하고, 순위별 근거는 배치 맥락에 맞게 새로 작성한다(§6). **특정 대분류**(Phase 0.5에서 확정, `특정전환` 포함)는 해당 부서 맥락만 반영해 독립적으로 추천한다.
6. ⑥ 추천 결과 자기검증 — Task로 **result-validator** 호출. 실패 시 driver-recommender 재호출 최대 2회, 이후도 실패면 에스컬레이션
7. ⑦ 배치 병합 및 산출물 출력 (코드) — `batch-tracker`의 `track_batch.py complete`로 `accounts_master.json`의 `categories[대분류]` 상태 갱신, `excel-io`의 `write_results.py`로 회계사 확정 UI(§6)가 포함된 `batch_{id}-{부서}_final.xlsx` 생성/append
8. ⑧ 데모 요약 리포트 생성 — 메인 세션이 직접 작성, `/docs/demo_summary_{batch_id}.md` (Phase 0.5 분리 결과 요약 포함)

**회계사 확정 피드백 루프** (Phase 0.5 이후, 상시 — 경로 A/B 둘 다 최종적으로 `accounts_master.json`의 `회계사확정`·`처리경로`·(해당 시)`four_type`/`원가동인`으로 수렴한다)

추가판단 필요 계정 해소는 공통적으로 두 가지 방식을 지원한다:
- **Path 1**: 4-type만 확정 → 원가동인은 아직 비워둠 → driver-recommender 재호출부터 다시 필요(**⑤~⑦ 재실행**), `처리경로`="AI추천-확정(대기)".
- **Path 2**: 4-type과 원가동인을 함께 확정 → AI 재추천 없이 그 값 그대로 반영, **result-validator만 재실행**, `처리경로`="사람직접확정".

**경로 A — 부서별 엑셀(`batch_{id}_final.xlsx`)**: **"추가판단 필요" 대분류의 Path 1/Path 2 해소만 지원한다** (명확 대분류의 승인/수정 확정은 다루지 않는다 — 그건 경로 B를 쓴다). 회계사가 "4-type 분류(사람 확정)"/"확정 여부"/"확정 원가동인"을 채워 재업로드한 엑셀을 `read_confirmations.py`(행 단위 파싱, 대분류 스키마)로 읽고, `track_batch.py confirm(master_path, excel_path)`이 accounts_master.json 기준으로 **아직 `카테고리분류상태: "추가판단필요(검토대기)"`인 대분류만** 반영한다 — 이미 다른 경로로 해소됐거나 엑셀에 여전히 값이 비어있는 대분류는 건드리지 않는다(전체 재처리 금지). 같은 대분류의 여러 계정코드 행에 값이 서로 다르면 반영하지 않고 '이상'으로 보고한다. 반환값(`재실행_필요_recommend`/`재실행_필요_validate_only`/`이상`)의 의미는 경로 B와 동일하다.

**경로 B — Streamlit 확정 앱(`streamlit_app/app.py`)**: 회계사가 "전체" 탭에서 대분류를 확정하고(필요 시 세부계정을 예외 지정) 앱 내 "전체 저장" 버튼을 누르면, 앱이 아래 순서를 코드로 직접 수행한다 (오케스트레이터가 별도로 개입할 필요는 없으나, 결과 파일이 갱신되는 절차는 동일하다).
1. 앱이 화면상의 확정 초안(`categories` + `sub_accounts` 두 배열)을 `confirmed_results.json`으로 저장한 뒤, `batch-tracker`의 `track_batch.py apply-browser-confirmations`(함수 호출)로 이를 `accounts_master.json`에 반영한다. **대분류 확정은 부서 무관 단일 결정이라 등장부서 전체에 자동 전파**되고, 세부계정(계정코드) 중 "예외 지정"된 것만 `accounts[code].예외확정`에 개별 저장된다 — 계정코드도 전사 공통 코드 체계라 부서 무관 단일 레코드이므로, 예외 지정하면 그 계정이 등장하는 모든 부서에 동일하게 적용된다.
2. 반환된 재작업 목록(`재실행_필요_recommend`/`재실행_필요_validate_only`)에 따라 ⑤~⑦ 또는 result-validator만 재실행해야 한다 — 이 재실행 자체는 오케스트레이터(서브에이전트 호출)가 담당하므로, 앱 저장 후 반환된 재작업 목록을 사용자에게 공유받아 처리한다. 목록의 항목은 이제 계정코드가 아니라 대분류명(또는 예외지정 세부계정은 계정코드)이다.
3. 앱은 저장 직후 각 부서의 `batch_{id}-{부서}_final.xlsx`를 `write_results.write_workbook(..., overwrite=True)`로 즉시 재생성한다 — `category_utils.resolve_account_effective()`로 원칙준용/예외지정을 반영한 최종값을 실어 보내므로 "미확정" 기본값이 아니라 이미 확정된 상태로 재생성된다. 다만 재작업(⑤~⑦ 재실행)이 필요한 대분류는 이 재생성만으로 끝나지 않고, 재실행 이후 다시 반영해야 최신 AI 추천이 엑셀에 실린다.
4. 사람 확정 4-type을 아직 입력하지 않은 추가판단 필요 대분류는 조치하지 않는다(앱에서 "저장" 자체가 걸리도록 되어 있다). `이상`(예: "확정"을 선택했는데 확정 원가동인이 비어있음)은 사용자에게 보고 후 지시를 기다린다.

## 3. 서브에이전트 호출 규칙

- 서브에이전트는 메인 세션만 Task로 호출한다. 서브에이전트끼리는 서로 호출할 수 없다(설계·플랫폼 제약).
- 데이터는 항상 **파일 경로**로 전달한다 (`/output/batch_{id}_*.json`). 원본 데이터를 프롬프트에 인라인으로 붙여넣지 않는다.
- 호출 시 입력 파일 경로와 출력 파일 경로를 명시적으로 지시한다.
- 앞 단계 산출물이 검증을 통과하기 전에는 다음 단계를 호출하지 않는다 (예: classifier 결과 검증 전 recommender 호출 금지).
- account-classifier는 Phase 1의 4-type 분류뿐 아니라 Phase 0.5의 "재확인 모드"(공통/특정 여부 재확인)로도 호출될 수 있다. 호출 시 어느 모드인지 명시하고, 출력 파일 경로(`llm_verdicts.json`)를 지정한다.

## 5. 에스컬레이션 프로토콜

사용자 확인이 필요한 상황에서는 절대 임의로 판단해 넘어가지 않고 반드시 응답을 기다린다.

- **부서 파일 누락** (Phase 0 전제조건): 전체 부서 목록 대비 누락된 부서를 나열하고 확인 요청
- **공통/특정 분리 추가판단 필요** (Phase 0.5): 규칙 기반 1차 판별(세부계정 등장부서 분포 이질성 또는 부서별 금액 쏠림)로 확정하지 못한 대분류는 account-classifier 재확인을 거치되, 재확인 결과도 판정 근거가 불충분하면(예: 부서별 맥락 정보 자체가 부족) 임의로 "공통유지"/"특정전환"을 강행하지 않고 사용자에게 대분류와 플래그 사유 상세를 제시해 확인을 요청한다.
- **입력 검증 반복 실패**: 어떤 컬럼/형식이 문제인지 구체적으로 제시하고 원본 확인 요청
- **추가판단 필요 계정**: 4-type을 임의 배정하지 않는다. `batch_{id}_final.xlsx`의 "4-type 분류"(AI) 컬럼에 "추가판단 필요"로 표시하고, "추천 근거" 컬럼에 계정 성격 설명(정의/추정 근거/왜 추가판단이 필요한지)만 담는다. AI는 이 상태에서 멈추고, 회계사가 "4-type 분류(사람 확정)" 컬럼에 값을 입력하기 전까지 원가동인 추천 단계로 넘기지 않는다 — 4-type만 입력(Path 1)했는지, 원가동인까지 함께 입력(Path 2)했는지는 §2·§6에 따라 각각 다르게 처리한다. 이 계정 때문에 나머지 배치 처리를 막지는 않는다.
- **추천 검증 반복 실패** (자동 재시도 2회 초과): 어떤 계정에서 어떤 이유로 실패했는지 보고 후 사용자 지시 대기
- **회계사 확정 이상** (`read_confirmations.py`의 `이상` 분류 — 예: "수정"/"확정"을 선택했는데 확정 원가동인이 비어있음, "4-type 분류(사람 확정)"에 알 수 없는 값 입력, 확정 원가동인은 입력했는데 "확정 여부"를 `확정`으로 선택하지 않아 Path 1/2 중 무엇을 의도했는지 불명확한 경우): 임의로 추정해 반영하지 않고 어떤 계정에서 어떤 값이 문제인지 구체적으로 보고 후 재확인 요청

## 6. 산출물 저장 규칙

- 모든 산출물은 `/output` 아래에만 저장한다.
- `accounts_master.json`의 처리상태(`처리완료`)는 **항목(계정) 단위로 즉시 갱신**한다. 배치 종료 시점까지 미루지 않는다 — 중단 후 재개(resume)의 유일한 기준이기 때문.
- 재개 로직: 처리가 중단되면 `accounts_master.json`에서 처리상태가 "완료"가 아닌 항목부터 재개한다. 처음부터 재처리하지 않는다. 이 재개 기준은 AI 추천 파이프라인 전용이며, 회계사 확정 진행 여부(`회계사확정`)와는 독립적이다.
- **Phase 0.5 산출물**: `account_segmentation.json`(원본)과 `account_segmentation.xlsx`(읽기 전용 참고본)를 함께 저장한다. `needs_llm_recheck`가 해소되기 전까지는 미완료 상태임을 산출물 자체에도 남긴다(`write_segmentation_excel.py`가 경고 출력).
- **`confirmed_results.json`**: Streamlit 앱이 "전체 저장" 시 자동으로 `/output`에 저장하는 파일이며, `track_batch.py apply-browser-confirmations`의 입력이 된다. 이 파일 자체는 산출물이 아니라 중간 입력 자료이므로 덮어쓰기·버전 관리에 유의하지 않아도 되지만, 반영 후에는 `accounts_master.json`이 유일한 진실 소스가 된다.
- 확정 UI는 `/streamlit_app/app.py`(Streamlit 앱, `streamlit run streamlit_app/app.py`로 실행)이 담당한다. 화면상의 확정 초안은 `st.session_state`에만 있다가 "전체 저장" 버튼을 눌러야 `accounts_master.json`과 부서별 최종 엑셀에 실제로 반영된다. 앱을 재시작해도 `accounts_master.json`의 `회계사확정`을 다시 읽어 이전 확정 상태를 복원한다.
- **최종 엑셀(`batch_{id}_final.xlsx`)에는 AI 추천 원가동인(최대 3순위)과 순위별 추천 근거를 그대로 노출한다.** (과거 버전의 "AI 판단 표시 금지" 원칙에서 변경됨 — 회계사가 승인/수정을 판단하려면 AI가 무엇을, 왜 추천했는지 봐야 하기 때문. 대신 "확정 여부"/"확정 순위" 컬럼으로 AI 추천과 회계사 확정을 항상 명확히 구분한다.)
  - 컬럼(정확한 순서·이름은 `write_results.py`의 `COLUMNS_EDITABLE`이 최종 소스): **대분류** / 계정코드 / 계정명 / **계정 설명**(account-classifier가 생성, 정보 표시용 잠금) / 부서 / 4-type 분류 / **판단 경로**(`AI 판단`\|`규칙 기반 폴백`\|`사람 직접 확정`, 정보 표시용 잠금 — `accounts_master.json`의 `categories[대분류].판단경로`/`accounts[code].예외확정.판단경로` 참조) / **추천 1순위 / 근거 1순위 / 추천 2순위 / 근거 2순위 / 추천 3순위 / 근거 3순위** / 4-type 분류(사람 확정) / **확정 순위** / **원가동인 적용 방식**(원칙준용/예외지정, 정보 표시용) / 확정 여부 / 확정 원가동인 / 비고. 후보가 3개보다 적으면 해당 순위 컬럼은 비워둔다(억지로 채우지 않음).
  - **명확 계정**: "확정 순위"는 `1순위`/`2순위`/`3순위`(존재하는 순위만)/`직접입력` 드롭다운(기본값 `1순위`). "확정 여부"는 `미확정`/`승인`/`수정` 드롭다운(기본값 `미확정`). **회계사가 반드시 1순위를 확정할 필요는 없다** — "확정 순위"에서 2·3순위를 고르고 "승인"하면 그 순위의 추천값이 확정된다. "확정 원가동인"은 승인 시 "확정 순위"에 해당하는 순위의 추천값을 자동 유지하는 수식(중첩 IF)이 기본이며, 회계사가 직접 입력(수정)하면 수식을 덮어써 확정값이 된다. `accounts_master.json`의 `회계사확정.확정순위`에 몇 순위를 확정했는지 기록해 나중에 "AI 1순위 추천 정확도"를 검토할 근거로 쓴다. "4-type 분류(사람 확정)"는 해당 없음(잠금).
  - **추가판단 필요(4-type 미확정) 계정도 같은 시트에 포함**한다. "4-type 분류"(AI)="추가판단 필요", AI 추천 순위 컬럼은 모두 공란(추가판단이 필요한 사유 설명은 "근거 1순위" 칸에 담는다). 이 계정은 명확 계정과 동일한 확정 컬럼 구조를 갖되 두 경로로 해소된다:
    - **Path 1**: "4-type 분류(사람 확정)"만 채움(확정 여부는 `미확정` 유지) → 4-type만 확정된 상태로 남아 원가동인 추천 재실행(⑤~⑦) 대상이 됨.
    - **Path 2**: "4-type 분류(사람 확정)" + "확정 원가동인"을 함께 채우고 "확정 여부"를 `확정`으로 선택 → AI 재추천 없이 사람이 직접 확정한 값 그대로 반영, result-validator만 재실행.
    - 추가판단 필요 계정의 "확정 여부" 드롭다운은 `미확정`/`확정` 2개 옵션만 갖는다(명확 계정의 `승인`/`수정`과 다름).
    - 어느 경로를 택했는지는 `accounts_master.json`의 `처리경로`(`AI추천-확정(대기)` = Path 1, `사람직접확정` = Path 2, `AI추천-확정` = 명확 계정의 통상 경로)로 구분해서 기록하고, 데모 리포트(⑧)에서 이 값을 인용해 AI-사람 역할 분리를 보여준다.
  - 시트 보호를 걸어 "확정 여부"/"확정 원가동인"(+ 추가판단 필요 계정 행의 "4-type 분류(사람 확정)")만 편집 가능하고 나머지(AI 산출물)는 잠근다 — 회계사가 실수로 AI 산출물을 고치지 못하게 한다.
  - **"확정 여부"가 "미확정"인 상태인 계정(추가판단 필요 계정의 Path 1 미해소 포함)은 데모 요약 리포트(⑧)에서 "최종 확정" 사례로 카운트하지 않는다.** "AI 추천 단계"로만 구분해서 보고한다(AI-사람 역할 분리를 명확히 보여주기 위함). "승인"/"수정"/"확정"만 확정 사례로 집계한다.
- **공통 대분류(Phase 0.5에서 확정)는 전체 부서에 통일된 원가동인을 원칙**으로 하고, `categories[대분류].회계사확정`은 **부서 무관 단일 객체**로 기록한다(계정코드 체계였던 과거와 달리 부서별 dict가 아님). 부서마다 다르게 하고 싶으면 그 세부계정(계정코드)을 "예외 지정"해 `accounts[code].예외확정`에 개별 기록한다 — 계정코드도 전사 공통 코드 체계라 부서 무관 단일 레코드이므로, 예외 지정하면 그 계정이 등장하는 모든 부서에 동일하게 적용된다. 이것이 원칙/예외 UI의 데이터 모델이다.
- **세부계정 설명**: account-classifier가 대분류 분류와 함께 세부계정(계정코드)별로 어떤 성격의 비용인지 1~2문장 설명(`세부계정_설명`)을 생성하고, `accounts[code].설명`에 저장한다. 참조 문서에 근거가 있으면 인용하고("문서인용"), 없으면 계정명 기반 추정임을 명시한다("일반적인 회계 관행 기반 추정"). 확정 UI(세부계정 카드)와 최종 엑셀("계정 설명" 컬럼)에 그대로 노출한다.
- **특정 대분류**(Phase 0.5에서 원래 특정이었거나 `특정전환` LLM 판정을 받은 경우)는 해당 부서 맥락만 반영해 독립적으로 추천·확정한다. 다른 부서의 확정 결과를 참조하지 않는다.
- 전체 스키마의 최종 소스는 `.claude/skills/batch-tracker/references/accounts_master_schema.md`다 — 이 문서와 실제 필드가 어긋나면 스키마 문서를 갱신한다.

## 7. 참조 문서 및 근거 인용 규칙

- `cost-driver-framework` 스킬의 참조 문서(ABC 원가계산 원칙, 4-type 분류체계, 보험회계 일반 지식)를 판단·근거 작성의 1차 소스로 사용한다. 이 문서들은 공개된 관리회계·보험회계 일반론을 바탕으로 재구성한 것이며, 특정 프로젝트·고객사의 실사 자료를 포함하지 않는다.
- 근거 문장은 참조 문서를 실제로 인용해야 한다. 문서에서 근거를 찾을 수 없으면 "일반 회계 지식 기반 추정"임을 명시한다. 그럴듯하지만 지어낸 근거를 쓰지 않는다.
